# 调度整体使用说明

本文只对应 `免费版模块/5.6` 当前磁盘上的真实实现整理，围绕当前公开仓库实际保留的能力展开说明。

这份文档的目标是把免费版从安装、启动、配置到线程调度主线一次讲清，方便后续自己维护或公开发布。

## 1. 免费版当前保留什么

当前免费版保留的能力只有这些：

1. CPU 调速器切换
2. CPU 最大频率和最小频率设置
3. governor 参数写入
4. `cpuset` 配置
5. `cpuctl` 配置
6. 场景提频
7. 自定义线程调度

当前免费版不包含这些能力：

1. 设备授权相关逻辑
2. 当前仓库未维护的实验性频率策略链
3. 当前仓库未保留的动态调节能力

所以后续维护时，重点看 CPU 模式、场景分类和线程专项三条主线，不要再按历史目录思路去找当前仓库没有保留的配置。

## 2. 当前目录结构

免费版目录里最关键的文件和目录如下：

| 路径 | 作用 |
| :- | :- |
| `module.prop` | 模块版本信息 |
| `customize.sh` | 刷入阶段脚本 |
| `service.sh` | 开机入口 |
| `activity_diaodu.rc` | 拉起主二进制的脚本 |
| `bin/activity_diaodu` | 设备侧实际运行的主程序 |
| `bin/activity_diaodu.cpp` | 主程序源码入口 |
| `bin/activity_mode_watch.inc` | 模式配置与热重载逻辑 |
| `bin/activity_scene_foreground.inc` | 前台应用识别 |
| `bin/activity_scene_policy.inc` | 场景分类与场景策略 |
| `bin/activity_scheduler_internal.inc` | 线程规则、绑核、优先级逻辑 |
| `bin/activity_scheduler_runtime.inc` | 线程运行时 |
| `bin/cpu/<SoC>.json` | 模式级 CPU 配置 |
| `bin/cpu/<拓扑>.json` | 线程专项配置 |
| `config/categories.json` | 场景分类配置 |
| `线程源码/` | eBPF 相关源码与构建产物 |

## 3. 安装链和启动链

### 3.1 刷入时做了什么

`customize.sh` 当前主要做这些事：

1. 检查基础系统文件是否存在
2. 创建 `config/mode.txt`
3. 预写默认模式和常见游戏包名模式覆盖
4. 屏蔽 `/system/vendor/bin/msm_irqbalance`
5. 设置模块目录权限
6. 执行 `vtools/init_vtools.sh`

也就是说，免费版安装后最先生成的核心配置之一就是：

```txt
/data/adb/modules/muronggameopt/config/mode.txt
```

### 3.2 开机启动顺序

`service.sh` 当前启动顺序如下：

1. 等待系统启动完成
2. 执行 `vtools/init_vtools.sh`
3. 写入少量系统节点
4. 后台执行 `activity_diaodu.rc`
5. `activity_diaodu.rc` 再拉起 `bin/activity_diaodu`

所以真正持续工作的核心是：

```txt
bin/activity_diaodu
```

## 4. 免费版配置主线

免费版当前最重要的配置文件是这 4 类：

1. `config/mode.txt`
2. `bin/cpu/<SoC>.json`
3. `bin/cpu/<拓扑>.json`
4. `config/categories.json`

你可以这样记：

1. 模式和频率改 `bin/cpu/<SoC>.json`
2. 线程专项改 `bin/cpu/<拓扑>.json`
3. 场景分类改 `config/categories.json`
4. 当前模式和应用级模式覆盖改 `config/mode.txt`

## 5. `mode.txt` 是做什么的

`mode.txt` 不只是保存一个当前模式，它的真实结构是：

```txt
powersave
com.tencent.tmgp.sgame balance
com.miHoYo.Yuanshen balance
```

含义如下：

1. 第一行是全局默认模式
2. 后面每一行是 `包名 模式`
3. 当前前台应用如果命中包名覆盖，就优先使用对应模式
4. 没命中时，回到第一行的全局默认模式

当前免费版支持的模式值仍然是：

1. `powersave`
2. `balance`
3. `performance`
4. `fast`

## 6. `bin/cpu/<SoC>.json` 管什么

这类文件例如：

1. `bin/cpu/SM8650.json`
2. `bin/cpu/SM8750.json`
3. `bin/cpu/SM8850.json`

它们负责每个模式下的 CPU 基础地基，包括：

1. `cpu_policies`
2. `cpuset`
3. `cpuctl`

免费版这里最该关注的就是：

### 6.1 `cpu_policies`

主要控制：

1. 每个簇的 `governor`
2. `max_freq`
3. `min_freq`
4. `params`

常见结构类似：

```json
{
  "cluster": 0,
  "governor": "walt",
  "max_freq": 2227200,
  "min_freq": 384000,
  "params": {
    "hispeed_freq": "960000"
  }
}
```

### 6.2 `cpuset`

主要控制这些组的 CPU 范围：

1. `background`
2. `systembackground`
3. `foreground`
4. `topapp`

这是模式级系统线程组分配地基，不是精细线程专项。

### 6.3 `cpuctl`

当前免费版仍支持：

1. `uclamp_min`
2. `uclamp_max`
3. `latency_sensitive`
4. `shares`

这部分决定不同线程组的 CPU 利用率上下限和时间片权重。

## 7. `bin/cpu/<拓扑>.json` 管什么

这类文件例如：

1. `bin/cpu/6+2.json`
2. `bin/cpu/6+1.json`
3. `bin/cpu/4+4.json`

这类文件当前就是免费版线程专项主线。

免费版这里已经按你的要求裁掉了一批不相关结构，维护时重点只看：

1. `apps`
2. `threads.main`
3. `threads.gfx`
4. `threads.render`
5. `threads.worker`
6. `threads.other`
7. `threads.custom`
8. `default_action`

### 7.1 线程专项怎么理解

可以把它理解成：

1. 先按应用匹配规则
2. 再识别该应用里的关键线程
3. 最后给这些线程设置绑核、优先级和调度策略

### 7.2 `custom` 是最常用的入口

如果你想给某个特定线程单独绑核，通常就是在：

```json
threads.custom
```

里加一条规则。

常见结构：

```json
{
  "name": "主渲染副核心",
  "selectors": [],
  "limit": 1,
  "action": {}
}
```

### 7.3 `action` 能做什么

当前免费版实际会吃这些字段：

1. `clusters`
2. `cpus`
3. `mask_hex`
4. `nice`
5. `scheduler`

一般建议：

1. 优先用 `clusters`
2. 非必要不要写死 `cpus`
3. `fifo` 和高 RT 优先级只给极少数关键线程

## 8. `config/categories.json` 管什么

这个文件负责场景分类和场景默认策略。

它的作用不是改线程名，而是：

1. 识别当前前台属于哪一类
2. 给这一类应用套默认策略
3. 让免费版在不同场景下切不同的基础倾向

常见分类来源有：

1. `packages`
2. `activities`
3. `processes`

典型场景包括：

1. 游戏
2. 桌面
3. 相机
4. 扫码
5. 小程序

这层更像“场景级默认规则”，而线程拓扑文件更像“应用级精细规则”。

## 9. 当前运行时大致怎么工作

当前免费版运行时可以这样理解：

1. 启动后读取 `mode.txt`
2. 根据当前设备型号加载对应 `bin/cpu/<SoC>.json`
3. 根据当前拓扑加载 `bin/cpu/<拓扑>.json`
4. 读取 `config/categories.json`
5. 识别前台应用
6. 确定当前模式
7. 应用 CPU、`cpuset`、`cpuctl`
8. 启动线程调度运行时，对前台应用线程进行识别和绑核

## 10. 哪些文件最值得改

如果只是日常二改，通常只需要动下面几类文件：

### 10.1 改模式频率和 governor

改：

```txt
bin/cpu/<SoC>.json
```

### 10.2 改应用线程绑核

改：

```txt
bin/cpu/<拓扑>.json
```

### 10.3 改场景识别

改：

```txt
config/categories.json
```

### 10.4 改默认模式或应用模式覆盖

改：

```txt
config/mode.txt
```

## 11. 构建方式

免费版当前本地构建链是两步：

### 11.1 先生成 eBPF 产物

在 `线程源码/` 目录执行：

```bat
编译monitor.skel.h.bat
```

生成：

1. `线程源码/monitor.bpf.o`
2. `线程源码/monitor.skel.h`

### 11.2 再编译主程序

在 `bin/` 目录执行：

```bat
build.bat
```

生成：

1. `bin/activity_diaodu`
2. `bin/monitor.bpf.o`
3. `bin/libc++_shared.so`

## 12. 当前免费版维护建议

后续维护这套免费版时，建议记住下面这些原则：

1. 不要把当前仓库没有保留的旧逻辑重新混回公开版主线
2. 想改模式和频率，优先看 `bin/cpu/<SoC>.json`
3. 想改应用线程，优先看 `bin/cpu/<拓扑>.json`
4. 想改场景判断，优先看 `config/categories.json`
5. 想改应用默认模式，优先看 `config/mode.txt`
6. 改了 `线程源码/monitor.bpf.c` 后，要先重新生成 `monitor.skel.h`
7. 改了 `bin` 下 C/C++ 代码后，要重新执行 `build.bat`

## 13. 一句话记忆

如果只记一句话：

1. 免费版主线就是 `mode.txt + SoC JSON + 拓扑 JSON + categories.json + activity_diaodu`

这 5 条串起来，就是当前免费版 5.6 的完整调度主线。
